### Fetching Scalar Fields

Scalar fields refer to all non-associative fields in a database table. Taking fetching the book name as an example:

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java

BookTable book = BookTable.$;

List<Book> books = sqlClient
    .createQuery(book)
    .select(
        book.fetch(
            // highlight-next-line
            BookFetcher.$.name()
        )
    )
    .execute();

```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin

val books = sqlClient
    .createQuery(Book::class) {
        where(table.edition.eq(3))
        select(
            table.fetchBy { 
                // highlight-next-line
                name()
            }
        )
    }
    .execute()

```

</TabItem>
</Tabs>

:::note
For Java, the Annotation processor will automatically generate a Fetcher class for each entity interface, which is `BookFetcher` in this example.
:::

The generated SQL is:

```sql
select 
    tb_1_.ID, 
    /* highlight-next-line */
    tb_1_.NAME  
from BOOK as tb_1_
where tb_1_.EDITION = ?
```

:::note 
The Java code does not call the `id()` method of BookFetcher, but we can see the SQL statement still queries the id property of the object.

The id property is treated specially and will always be queried, not controlled by object fetchers. 

In fact, the auto-generated BookFetcher class does not even have an `id()` method, because it is not needed.
:::

The printed result is (original output is compact, formatted here for readability):

```json
[
    {
        "id":3,
        // highlight-next-line
        "name":"Learning GraphQL"
    },
    ...Other objects omitted...  
]
```

### Fetching Multiple Fields

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java
BookTable book = BookTable.$;

List<Book> books = sqlClient
    .createQuery(book)
    .select(
        book.fetch(
            BookFetcher.$
                // highlight-next-line
                .name()
                // highlight-next-line
                .edition()
        )
    )
    .execute();
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
val books = sqlClient
    .createQuery(Book::class) {
        where(table.edition.eq(3))
        select(
            table.fetchBy {
                // highlight-next-line
                name()
                // highlight-next-line
                edition()
            }
        )
    }
    .execute()
```

</TabItem>
</Tabs>

:::note
Object fetchers are immutable objects, each method call returns a new object fetcher. 

In the **Java** code above:
-   `BookFetcher.$`
-   `BookFetcher.$.name()`  
-   `BookFetcher.$.name().edition()`

are three different object fetchers, each immutable.

Object fetchers can be freely shared using static variables since they are immutable.
:::

The generated SQL is: 

```sql
select
    tb_1_.ID,
    /* highlight-next-line */  
    tb_1_.NAME,
    /* highlight-next-line */
    tb_1_.EDITION   
from BOOK as tb_1_
where tb_1_.EDITION = ?
```

The printed result is (original output is compact, formatted here for readability):

```json
[
    {
        "id":3,
        // highlight-next-line
        "name":"Learning GraphQL",
        // highlight-next-line
        "edition":1
    },
    ...Other objects omitted...
]
```

### Fetching All Scalar Fields

In some scenarios, fetching all non-associative fields is a very common operation, so `allScalarFields()` is provided to load all non-associative fields.

<Tabs groupId="language">
<TabItem value="java" label="Java">  

```java
BookTable book = BookTable.$;

List<Book> books = sqlClient
    .createQuery(book)
    .select(
        book.fetch(
            // highlight-next-line
            BookFetcher.$.allScalarFields()
        )
    )
    .execute(); 
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
val books = sqlClient
    .createQuery(Book::class) {
        where(table.edition.eq(3))
        select(
            table.fetchBy {
                // highlight-next-line
                allScalarFields()
            }
        )
    }
    .execute()
```

</TabItem>
</Tabs>


The generated SQL is:

```sql  
select
    tb_1_.ID,
    // highlight-start 
    tb_1_.NAME,
    tb_1_.EDITION,
    tb_1_.PRICE 
    // highlight-end
from BOOK as tb_1_
where tb_1_.EDITION = ?
```

The printed result is (original output is compact, formatted here for readability):

```json
{
    "id":3,
    // highlight-start
    "name":"Learning GraphQL", 
    "edition":3,
    "price":51.00
    // highlight-end
}
// Other objects omitted
``` 

:::info
`allScalarFields()` only loads non-associative fields. In the example we can see it queries `id`, `name`, `edition`, `price`, but does not query the associative properties `store` and `authors`. 

Fetching associative properties will be covered in [the next document](./association).
:::


### Negative Attributes (Exclude Unneeded Properties) 

The previously discussed attributes are positive attributes that incrementally add fields to query. But in some cases, we only need to exclude some specific fields, while querying all other fields.

This is when negative attributes can be used to exclude unneeded fields.

<Tabs groupId="language">
<TabItem value="java" label="Java">  

```java
BookTable book = BookTable.$;

List<Book> books = sqlClient
    .createQuery(book)
    .select(
        book.fetch(
            BookFetcher.$
            .allScalarFields()
            // highlight-next-line
            .edition(false)
        )
    )
    .execute();
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin 
val books = sqlClient
    .createQuery(Book::class) {
        where(table.edition.eq(3))
        select(
            table.fetchBy {
                allScalarFields()
                // highlight-next-line
                edition(false)
            }
        )
    }
    .execute()
```

</TabItem>
</Tabs>

`edition(false)` uses false as a parameter, making it a negative attribute.

-   The attributes of `allScalarFields()` are `id + name + edition + price`

-   `edition(false)` means `-edition`

So in combination, the finally fetched attributes are `id + name + price`

:::info
- For positive attributes, `edition()` and `edition(true)` are equivalent
- When most attributes need to be fetched and only a few excluded, using `allScalarFields` with negative attributes can be very useful
:::

The generated SQL is:

```sql
select
    tb_1_.ID,
    tb_1_.NAME, 
    tb_1_.PRICE    
from BOOK as tb_1_
where tb_1_.EDITION = ? 
```

The printed result is (original output is compact, formatted here for readability):

```json  
[
    {
        "id":3,
        "name":"Learning GraphQL",
        "price":51.00
        // `edition` is not here        
    },
    ...Other objects omitted...
]
```
