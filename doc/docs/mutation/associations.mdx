## Premise of This Article

Jimmer supports [triggers](./trigger), including BinLog Triggers and Transaction Triggers. 

If using Transaction Triggers, it will cause Jimmer modification operations to generate more complete query statements to simulate triggers.

The various scenarios discussed in this article and their respective generated SQL are all based on the premise of not using Transaction Triggers.

## Overview

First, the junction table is hidden by the object model, with no direct corresponding entity type. This problem has been explained in [Query Junction Table](../query/associations), so I won't repeat it here.

In addition, in the [Save Command](./save-command) section, we introduced the save command. It can compare the existing data in the database with the new data the user wants to save. If a relation attribute based on the junction table has changed, the junction table will be modified.

Admittedly, the Save Command feature is very powerful, and modifying the junction table is just a very small part of its capabilities. However, sometimes we just need a simple way to insert or delete data directly from the junction table, we don't need the full capabilities of the Save Command.  

Jimmer allows developers to directly insert and delete data from the junction table in a simpler way.

## Insert Association

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java
sqlClient
    .getAssociations(BookProps.AUTHORS)
    // highlight-next-line
    .save(12L, 3L); 
```

</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin 
sqlClient
    .getAssociations(Book::authors)
    // highlight-next-line
    .save(12L, 3L)
```

</TabItem>
</Tabs>

The generated SQL is:

```sql
insert into
    BOOK_AUTHOR_MAPPING(
        /* highlight-next-line */
        BOOK_ID, AUTHOR_ID  
    )
    values (?, ?)
```

This example shows how to use the association property `Book.authors` to operate the junction table.

Similarly, we can achieve the same goal through the association property `Authors.books`:

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java
sqlClient
    .getAssociations(AuthorProps.BOOKS) 
    // highlight-next-line
    .save(3L, 12L);
```

</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin
sqlClient
    .getAssociations(Author::books)
    // highlight-next-line 
    .save(3L, 12L) 
```

</TabItem>
</Tabs>

Note that compared to the previous example, the parameter order of the `save` method is different due to the different association direction.

The generated SQL is:

```sql
insert into
    BOOK_AUTHOR_MAPPING(
        /* highlight-next-line */
        AUTHOR_ID, BOOK_ID
    ) 
    values (?, ?)
```

It's easy to see that the order of the insert statement columns is also different.

## Reverse Conversion  

For bidirectional associations, `reverse` can be used to switch between forward and reverse associations:

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java 
Associations bookToAuthor = sqlClient
    .getAssociations(BookProps.AUTHORS);
Associations authorToBook = sqlClient
    .getAssociations(AuthorProps.BOOKS);
Associations authorToBook2 =
    // highlight-next-line
    bookToAuthor.reverse(); 
Associations bookToAuthor2 =
    // highlight-next-line
    authorToBook.reverse();
```

</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin
val bookToAuthor = 
    sqlClient.getAssociations(
        Book::authors
    )
val authorToBook = 
    sqlClient.getAssociations(
        Author::books  
    )
val authorToBook2 =
    // highlight-next-line
    bookToAuthor.reverse()
val bookToAuthor2 =
    // highlight-next-line 
    authorToBook.reverse()
```

</TabItem>
</Tabs>

In this code:
-   `bookToAuthor` and `bookToAuthor2` are completely equivalent. 
-   `authorToBook` and `authorToBook2` are completely equivalent.

## Bulk Insert  

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java 
sqlClient
    .getAssociations(BookProps.AUTHORS)
    // highlight-next-line
    .batchSave(
        Arrays.asList(
            new Tuple2<>(10L, 1L),
            new Tuple2<>(11L, 1L), 
            new Tuple2<>(12L, 1L),
            new Tuple2<>(10L, 3L),
            new Tuple2<>(11L, 3L),
            new Tuple2<>(12L, 3L)  
        )
    );
```

</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin
sqlClient
    .getAssociations(
        Book::authors  
    )
    // highlight-next-line 
    .batchSave(
        listOf(
            Tuple2(10L, 1L),
            Tuple2(11L, 1L),
            Tuple2(12L, 1L),
            Tuple2(10L, 3L),
            Tuple2(11L, 3L),
            Tuple2(12L, 3L)
        )
    ) 
```

</TabItem>
</Tabs>

Here there are 3 books, two authors, and 6 total combination ways. The `batchSave` method inserts all 6 combinations into the junction table, generating the following SQL:

```sql  
insert into BOOK_AUTHOR_MAPPING(
    BOOK_ID, AUTHOR_ID
) values
    /* highlight-start */
    (?, ?),
    (?, ?), 
    (?, ?),
    (?, ?),
    (?, ?),
    (?, ?) 
    /* highlight-end */
```

The above approach is very cumbersome. For `3 * 2 = 6` it is still acceptable, but what about `7 * 9 = 63`? Should we really construct 63 tuples? 

Jimmer-sql provides a shorthand syntax:

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java 
sqlClient
    .getAssociations(BookProps.AUTHORS)
    // Save cartesian product of two id sets  
    // highlight-next-line
    .batchSave(
        Arrays.asList(
            10L, 11L, 12L
        ),
        Arrays.asList(
            1L, 3L
        )
    );
```
</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin
sqlClient
    .getAssociations(
        Book::authors
    )
    // Save cartesian product of two id sets
    // highlight-next-line 
    .batchSave(
        listOf(10L, 11L, 12L),
        listOf(1L, 3L)
    )
```
</TabItem>  
</Tabs>

The above `batchSave` method takes two collection parameters and inserts the cartesian product of the two collections into the junction table, so its functionality is the same as the previous example.

## Check Existence

Inserting id tuples that already exist into the junction table will cause errors, as it violates uniqueness constraints.

To solve this problem, existence checks can be performed.

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java
sqlClient
    .getAssociations(BookProps.AUTHORS)
    .batchSaveCommand(
        Arrays.asList(
            10L, 11L, 12L
        ), 
        Arrays.asList(
            1L, 3L
        )
    )
    // highlight-next-line
    .checkExistence()
    .execute();
```
</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin 
sqlClient
    .getAssociations(
        Book::authors
    )
    // highlight-next-line
    .batchSave(
        listOf(10L, 11L, 12L),
        listOf(1L, 3L),
        // highlight-next-line
        checkExistence = true
    )
```

</TabItem>
</Tabs>

Here `checkExistence()` indicates that it will first check which data to be inserted already exists, and then only insert data that does not exist.

The generated SQL statements are:

1.  Check which data to be inserted already exists:

    ```sql
    select
        BOOK_ID, AUTHOR_ID
    from BOOK_AUTHOR_MAPPING
    where
        (BOOK_ID, AUTHOR_ID) in(
            (?, ?),
            (?, ?), 
            (?, ?)
        )
    ```

2.  Using the results of the previous query, the data that really needs to be inserted can be calculated. Skip this step if no data needs to be inserted; otherwise, execute this step.

    Here, assuming it is determined that only two rows of data really need to be inserted, the generated SQL is:

    ```sql 
    insert into BOOK_AUTHOR_MAPPING(
        BOOK_ID, AUTHOR_ID  
    ) values
        (?, ?),
        (?, ?)
    ```

## Notes  

:::caution
Please do not abuse this feature. This feature is only suitable for simple insertion and deletion of junction tables. It is recommended to only use it when facing the simplest incremental operations.  
:::

Let's look at an example. If comparison with existing data in the database is needed, this feature is obviously not the optimal choice, and the code will be very cumbersome, as follows:

<Tabs groupId="language">
<TabItem value="java" label="Java">  

```java title="Inappropriate Usage Example" 
public void setAuthors(long bookId, Collection<Long> authorIds) {

    // Get old author ids
    AssociationTable<Book, BookTableEx, Author, AuthorTableEx> table =
                AssociationTable.of(BookTableEx.class, BookTableEx::authors);    
    List<Long> oldAuthorIds = sqlClient
            .createAssociationQuery(table)
            .where(table.source().id().eq(1L))
            .select(table.target().id())
            .execute();

    // Delete `oldAuthorIds - authorIds`
    List<Long> detachedAuthorIds = new ArrayList<>(oldAuthorIds);
    detachedAuthorIds.removeAll(authorIds);
    sqlClient.getAssociations(BookProps.AUTHORS).batchDelete(
            Collections.singleton(bookId),
            detachedAuthorIds
    );

    // Insert `authorIds - oldAuthorIds`
    List<Long> attachedAuthorIds = new ArrayList<>(authorIds);
    attachedAuthorIds.removeAll(oldAuthorIds);
    sqlClient.getAssociations(BookProps.AUTHORS).batchSave(
            Collections.singleton(bookId),
            attachedAuthorIds
    );
}
```
</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin title="Inappropriate Usage Example"
fun setAuthors(bookId: Long, authorIds: Collection<Long>) {
    // Get old author ids
    val oldAuthorIds = sqlClient.queries.forList(Book::authors) {
        where(table.source.id eq bookId)
        select(table.target.id)
    }.execute().toSet()

    // Delete `oldAuthorIds - authorIds`
    sqlClient
        .getAssociations(Book::authors)
        .batchDelete(listOf(bookId), oldAuthorIds - authorIds)

    // Insert `authorIds - oldAuthorIds` 
    sqlClient
        .getAssociations(Book::authors)
        .batchSave(listOf(bookId), authorIds - oldAuthorIds)
}
```

</TabItem>
</Tabs>

It is obvious that dealing with it this way is very cumbersome.  

:::tip
When the business scenario is no longer simple addition or deletion, [Save Command](./save-command) is a better choice, as follows:
:::

<Tabs groupId="language">
<TabItem value="java" label="Java">

```java title="Better Usage Example"
public void setAuthors(long bookId, Collection<Long> authorIds) {
    sqlClient.save(
        BookDraft.$.produce(book -> {
            book.setId(bookId);
            book.setAuthorIds(authorIds);
        })
    );
}
```
</TabItem>

<TabItem value="kotlin" label="Kotlin">

```kotlin title="Better Usage Example"  
fun setAuthors(bookId: Long, authorIds: Collection<Long>) {
    sqlClient.save(
        new(Book::class).by {
            id = bookId
            authorIds = authorIds
        }
    )
}
```

</TabItem>
</Tabs>
